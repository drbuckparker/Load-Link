Why skip Path A (separate mirror DB + real sync)
This sounds principled but it doesn't actually buy you anything in your situation:

The companion server runs on Replit, not on the driver's device. So when a driver is in a cell dead zone, both loadlinklive.com and the companion API are equally unreachable — having a "local-first" mirror DB on Replit doesn't help.
Bidirectional sync is genuinely hard. Conflict resolution, soft-delete reconciliation, eventual consistency, idempotency for retries. The agent noted sync.ts has never actually run against two distinct DBs, so it's untested in its real role. That's months of work and a constant source of "why did this row get overwritten" bugs.
Joey is one person maintaining this. Adding a second database with a sync layer doubles the surface area he has to debug.
Path A makes sense only if you have a hard requirement that the companion server keeps working when the website's DB is down — and even then, you'd want a different solution (read replicas, cached responses) rather than full bidirectional sync.

Why I'm split between B and C, and what to do first
Both Path B (companion stays as a thin proxy) and Path C (collapse the companion server entirely) are reasonable end states. The decision depends on what the companion server actually does that the website doesn't.

If most of the companion's value is just "a different shape of API for mobile clients" — same data, slightly different routes — then you're paying the cost of two backends for a thin shim. Path C wins.

If the companion has meaningful mobile-specific logic (dashboard widget computation, calendar cascade, push-notification scheduling, anything that genuinely lives better near the mobile contract) and there are good reasons not to push that into the website routes, then Path B is fine.
