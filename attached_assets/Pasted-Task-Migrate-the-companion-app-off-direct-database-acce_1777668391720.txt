Task: Migrate the companion app off direct database access onto the existing Companion REST API.

Context
The companion app currently bundles a Postgres DATABASE_URL and connects directly to the production Neon database from the mobile client. This is a security problem (the DB password ships in the binary) and prevents a dev/prod split. The web project (https://loadink.replit.app) already exposes a Companion API that the mobile app should use instead.

Target architecture
Mobile app talks ONLY to the web project over HTTPS. No pg, no drizzle-orm, no Postgres connection in the mobile codebase.
Auth flow:
On app start (or login), call POST {API_BASE}/api/companion/auth/login with header X-API-Key: <COMPANION_API_KEY> and JSON body {"email": "<user_email>"} (password optional for social-login users). Response: { token, user }.
Save token in expo-secure-store.
On every subsequent request, send headers X-API-Key: <COMPANION_API_KEY> AND Authorization: Bearer <token>.
On any 401, clear the stored token and re-run login.
Health check endpoint for diagnostics: GET {API_BASE}/api/companion/health (just needs X-API-Key).
All other routes that the mobile app needs are the SAME routes the website uses (e.g., /api/jobs, /api/profile, /api/dashboard, /api/calendar/jobs, etc.). Just call them with the two auth headers.
Required environment variables (set via EAS / Expo config, NOT bundled as constants)
EXPO_PUBLIC_API_BASE_URL — https://loadink.replit.app for production builds, the dev Replit URL for development builds.
EXPO_PUBLIC_COMPANION_API_KEY — value will be provided by the user; same value for dev and prod is fine for now.
Configure these per build profile in eas.json:

{
  "build": {
    "development": { "env": { "EXPO_PUBLIC_API_BASE_URL": "<dev_replit_url>", "EXPO_PUBLIC_COMPANION_API_KEY": "<dev_key>" } },
    "production":  { "env": { "EXPO_PUBLIC_API_BASE_URL": "https://loadink.replit.app", "EXPO_PUBLIC_COMPANION_API_KEY": "<prod_key>" } }
  }
}

Steps
Build a single API client module (~50 lines) that handles login, token storage in expo-secure-store, header injection, and 401-retry.
Find every direct DB call in the codebase. Search for: pg, Pool, postgres://, drizzle, drizzle-orm, DATABASE_URL. For each one, replace with an apiClient.get/post/... call to the equivalent web route. If a needed route doesn't exist on the web side, list it and stop — do not bypass.
Remove pg, drizzle-orm, and any other direct-DB dependencies from package.json. Run npm install and confirm the lockfile updates.
Delete DATABASE_URL from the Expo / EAS Configurations panel entirely.
Verify zero remaining hits: rg -n "DATABASE_URL|pg\.|Pool|drizzle|postgres://" in the mobile app source should return zero matches in app code (test/script files in a separate folder don't count if you confirm with the user).
Run a development build (eas build --profile development or expo start against the dev URL) and verify login + at least one data screen works end-to-end.
Update the README with the new env-var setup.
Done criteria
package.json contains no Postgres driver.
Codebase has zero references to DATABASE_URL or direct DB clients.
App successfully logs in via /api/companion/auth/login and fetches the user's jobs over HTTPS.
Both development and production build profiles in eas.json have their own EXPO_PUBLIC_API_BASE_URL.
Things NOT to do
Do not add a new direct DB connection "just for one screen."
Do not hardcode the API key in source — only env vars.
Do not modify the web project (it's already done; just consume it).
If a needed endpoint is missing on the web side, stop and report the gap. The user will add it on the web side and resume.
