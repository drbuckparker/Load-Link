# Contractor notification when a driver applies to a job

A push notification alone does NOT populate the in-app notifications inbox or the home-screen bell badge. Both read from the `notifications` table (`GET /api/notifications`). Any "alert the user" feature must INSERT a notification row in addition to (or instead of) sending a push.

**Why:** the bell badge (`app/(tabs)/index.tsx`) and the notifications screen derive unread state from the `/api/notifications` query, which selects from the `notifications` table. `sendPushNotification` only hits Expo Push — it writes nothing to the DB. The driver-apply path historically sent a push but created no row, so contractors saw no bubble/inbox entry.

**Enum gotcha:** `notifications.type` is a USER-DEFINED enum `notification_type`, **owned by and shared with the website** (companion shares the website's Neon prod DB). Its values: new_load, load_accepted, load_approved, load_rejected, load_completed, message, general, foreman_invitation, job_expired, job_date_changed. There is NO `job_application` value — inserting it throws (the insert is wrapped in try/catch so it fails silently). Do NOT `ALTER TYPE ... ADD VALUE` on this shared enum unilaterally; reuse an existing value. The driver-apply notification reuses `new_load` (renders a briefcase icon + orange highlight, taps through to the job's applicants).

**Truck-horn sound (retargeted July 2026):** the horn now alerts DRIVERS to a NEW JOB in their area, NOT the contractor when a driver applies. Keyed off the PUSH payload's free-form `data.type === 'new_job'` (NOT the DB enum), checked in the foreground `addNotificationReceivedListener` in `app/_layout.tsx`. The contractor's driver-apply push reverted to the default sound. **Why:** the horn's product purpose is to make drivers apply fast to fresh jobs; a horn on application receipt was backwards.

**New-job driver alert is companion-owned, not website:** `POST /api/jobs` calls `notifyNearbyDriversOfNewJob()` which computes haversine distance from job origin to each driver's saved/last-known locations and, within their `search_radius_miles`, sends the truck-horn push + inserts a `new_load` inbox row. **Why companion-owned:** the website's `pushToWebsite('/api/jobs')` radius-notify was NOT delivering driver-facing rows for companion-posted jobs (no such rows in the shared DB for real recent jobs; only demo seed), and the user observed no new-job alert — so the companion must own it. **Known limitation:** jobs posted from the WEBSITE (not the companion) won't fire this horn; closing that requires a website-side change. Watch for duplicate pushes if the website later starts sending its own new-job push.

**Platform constraints:** the in-app expo-av horn listener (`app/_layout.tsx`) is iOS-only by design (returns early on android/web) — on Android the notification CHANNEL plays the sound at OS level instead, so no in-app duplicate is needed. A custom lock-screen/background push sound needs the sound bundled via the `expo-notifications` app.json plugin (`sounds: [...]`) AND a native rebuild (Expo Launch) — it does NOT work in Expo Go.

**Android notifications (enabled July 2026):** `lib/notifications.ts` no longer early-returns on Android. It registers a push token (passing the EAS `projectId` from `expo-constants` — REQUIRED on Android or `getExpoPushTokenAsync` throws) and creates two channels: `job-alerts` (MAX importance, vibration, sound `truckhorn.wav`) and `default` (HIGH, standard sound). **Android routes by CHANNEL, and the channel's importance/vibration/sound OVERRIDE the per-message values** — so the server must send `channelId: 'job-alerts'` for new-job pushes and `channelId: 'default'` otherwise (plus `priority: 'high'`); `channelId`/`priority` are ignored by iOS. **Delivery still requires FCM credentials** (google-services.json + FCM key uploaded to EAS) in the Android build — without it Android tokens fail silently and nothing is delivered.

**How to apply:** for any new "notify a user" event, insert a `notifications` row with a valid `notification_type` enum value; only add a push on top if a device alert is wanted. Pass any custom routing/sound hints through the push payload `data`, not the DB type.
